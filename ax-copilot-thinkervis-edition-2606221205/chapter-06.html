챕터 6. HTML MVP 병아리반 — 아이디어를 화면으로 보여주기

HTML MVP 실습은 코딩 수업이 아닙니다. 말로만 설명하던 아이디어를 한 화면으로 보여주고, 사람들이 같은 그림을 보며 이야기하게 만드는 수업입니다.

이 장을 시작하기 전에 보는 수업 지도

오늘 남길 것업무 아이디어를 입력, 처리, 출력 화면으로 나눈 HTML MVP 설계안
사람이 확인할 것예쁜 화면보다 사용자가 실제로 무엇을 입력하고 얻는지 확인한다.
강사 운영 포인트설명 20%, 시연 30%, 수강생 자기 업무 변환 50%로 운영합니다.

왜 HTML을 만들까

회의에서 “고객 이슈 대시보드가 필요합니다”라고 말하면 사람마다 다른 그림을 떠올립니다. 어떤 사람은 표를, 어떤 사람은 그래프를, 어떤 사람은 보고서를 생각합니다. Copilot에게 단일 HTML 초안을 만들게 하면 모두가 같은 화면을 보며 이야기할 수 있습니다.

단일 HTML 파일로 회의용 대시보드 초안을 만들어줘.
주제는 고객 VOC 우선순위 정리입니다.
포함할 것: 상단 요약 카드 3개, 문의 유형 TOP 5 표, 우선 조치가 필요한 항목, 다음 액션 목록.
디자인은 깔끔하고 업무용으로, 한 파일 안에 CSS까지 포함해줘.
실제 개인정보는 넣지 말고 샘플 데이터로 작성해줘.
[병아리] HTML을 모르는 사람도 할 수 있나요?
네. 이 실습의 목표는 HTML 문법을 외우는 것이 아닙니다. Copilot에게 원하는 화면을 설명하고, 나온 결과를 열어보며 업무에 맞게 수정 요청하는 것입니다.

수업 중 자주 생기는 문제

첫째, 코드가 너무 길어져서 수강생이 겁을 먹을 수 있습니다. 이때 강사는 “코드를 읽는 시간이 아니라 결과 화면을 보고 요구사항을 고치는 시간”이라고 말해야 합니다. 둘째, 브라우저에서 파일을 여는 방법을 모를 수 있습니다. 파일을 .html로 저장하고 더블클릭하면 된다고 천천히 안내하세요. 셋째, 디자인이 마음에 들지 않을 수 있습니다. 이때는 색상, 여백, 카드 수, 표 컬럼처럼 구체적으로 수정 요청하게 합니다.

tip

HTML MVP는 완성품이 아니라 대화용 초안입니다. 강사는 ‘이걸 배포하자’보다 ‘이 화면을 보고 어떤 의사결정을 할 수 있나요?’라고 물어야 합니다.

Thinkervis판 수업 설계 노트

이 장의 운영 목표는 수강생이 챕터 6. HTML MVP 병아리반 — 아이디어를 화면으로 보여주기를 “이해했다”고 말하는 데서 끝나지 않고, 자기 업무에 적용 가능한 산출물 하나를 남기는 것입니다. 따라서 강사는 설명, 데모, 개인 실습, 짝 토론, 공유 순서를 명확히 운영해야 합니다. 기능 설명이 10분을 넘으면 수강생의 손이 멈춥니다. 설명은 짧게, 실습은 작게, 피드백은 자주 주는 구조가 좋습니다.

Thinkervis판에서는 이 장을 “콘텐츠 전달”이 아니라 “수업 운영 단위”로 봅니다. 강사는 무엇을 설명할지보다 무엇을 남길지 먼저 정해야 합니다. 이 장의 핵심 산출물은 업무 아이디어를 화면 구성, 입력값, 출력값, 검증 흐름으로 정리한 HTML MVP 기획서입니다. 수강생이 이 산출물을 만들지 못했다면 설명을 많이 들었더라도 수업 목표는 아직 끝난 것이 아닙니다.

운영 원칙: 코드 결과물에만 집중하면 업무 문제 정의와 사용자의 실제 행동이 흐려집니다. 그래서 강사는 화면 시연, 개인 작업, 공개 피드백을 한 흐름으로 묶어야 합니다. “제가 보여드릴게요”에서 멈추지 말고 “이제 여러분 자료로 바꿔봅시다”까지 밀어주는 것이 중요합니다.
구간권장 시간강사 행동완료 기준
개념 열기5분업무 상황 하나로 문제를 제시한다.수강생이 왜 필요한지 말할 수 있다.
데모7분부족한 프롬프트와 개선 프롬프트를 비교한다.결과 품질 차이를 눈으로 확인한다.
개인 실습12분자기 업무로 바꾸게 한다.개인 프롬프트 초안 1개가 나온다.
공유6분2명 정도의 사례를 공개 피드백한다.다른 업무에도 적용 가능하다고 느낀다.

수업 전 준비

  • 자료 준비: 강사용 예제 1개와 수강생이 자기 업무로 바꿀 빈 템플릿 1개를 함께 준비합니다. 예제만 있으면 따라 쓰기에서 끝나고, 빈 템플릿만 있으면 초보자가 시작을 어려워합니다.
  • 환경 준비: Copilot 접속, 브라우저 로그인, 문서 붙여넣기 가능 여부를 수업 전 확인합니다. 실패 가능성이 있으면 같은 활동을 Chat 입력창, 문서 편집기, 메모장으로 대체하는 흐름을 미리 정합니다.
  • 시간 준비: 이 장에서 반드시 끝낼 활동 1개와 시간이 남을 때 할 확장 활동 1개를 분리합니다. 현장 수업은 늘 속도 차이가 나므로, 필수 활동을 작게 잡는 편이 전체 만족도가 높습니다.

강사용 진행 멘트

“지금부터는 기능을 많이 아는 사람이 이기는 시간이 아닙니다. 자기 업무를 작게 쪼개고, Copilot에게 맡길 부분과 사람이 확인할 부분을 구분하는 시간이 더 중요합니다. 화면이 저와 조금 달라도 괜찮습니다. 같은 원리로 목적, 자료, 결과물 모양, 주의사항을 넣어보면 됩니다.”
“결과가 마음에 들지 않으면 실패가 아니라 좋은 진단 자료입니다. Copilot이 이상하게 답했다면 대개 목적이 흐렸거나, 대상 독자가 빠졌거나, 하지 말아야 할 일이 빠진 경우입니다. 지금 나온 답을 보고 프롬프트를 고쳐보겠습니다.”

수강생 활동지 구성

활동지 칸수강생이 적을 내용강사가 보는 포인트
내 업무 한 줄반복해서 하는 실제 업무를 짧게 쓴다.너무 큰 업무라면 읽기, 정리, 초안, 검토 중 하나로 줄인다.
성공한 결과수업 끝에 손에 남아야 할 결과물을 쓴다.“잘 이해하기”가 아니라 파일, 문장, 표, 체크리스트처럼 확인 가능한 형태인지 본다.
AI에게 맡길 부분자료 정리, 초안 작성, 비교, 질문 생성 등 맡길 일을 고른다.판단, 승인, 책임 표현까지 통째로 맡기지 않도록 안내한다.
사람이 볼 기준사실 여부, 조직 규정, 민감정보, 톤, 누락 항목을 쓴다.검토 기준이 있어야 Copilot 결과를 업무물로 바꿀 수 있다.

평가 루브릭

  • 1점: Copilot에게 단순히 “해줘”라고만 입력한다.
  • 2점: 업무 목적이나 대상 중 하나를 포함한다.
  • 3점: 목적, 대상, 결과물 형식을 포함한다.
  • 4점: 검토 기준과 주의사항까지 포함한다.
  • 5점: Copilot 결과를 검토하고 추가 질문으로 개선한다.

피드백 방식

공개 피드백은 정답 맞히기처럼 운영하지 않습니다. 강사는 “좋다/나쁘다”보다 “더 선명해질 부분”을 말해야 합니다. 예를 들어 “대상 독자가 빠져서 말투가 흔들립니다”, “결과물 형식이 없어서 답이 길어질 수 있습니다”, “원문에 없는 추정을 금지하는 문장이 있으면 더 안전합니다”처럼 고칠 행동을 바로 알 수 있게 말합니다.

수강생 결과물을 화면에 띄울 때는 이름이나 실제 업무 정보가 노출되지 않도록 먼저 확인합니다. 민감한 사례는 강사가 즉석에서 기관명, 고객명, 금액, 일정 등을 가명으로 바꿔 읽어줍니다. Thinkervis판의 핵심은 실전성 높이기이지만, 실전 정보가 그대로 공개되는 순간 수업 신뢰가 무너질 수 있습니다.

Thinkervis 심화 운영 매뉴얼

이 심화 파트는 강사가 실제 강의안으로 바로 옮겨 적을 수 있도록 만든 운영 매뉴얼입니다. 같은 내용을 설명하더라도 초보 수강생, 업무 경험이 많은 관리자, 이미 AI를 써본 실무자는 서로 다른 지점에서 막힙니다. 강사는 “모두 같은 속도로 따라오기”를 목표로 삼기보다, 같은 산출물에 도달하는 여러 경로를 준비해야 합니다.

이 장의 중심 질문은 “챕터 6. HTML MVP 병아리반 — 아이디어를 화면으로 보여주기를 잘 설명했는가”가 아니라 “수강생이 수업 후 자기 업무에서 다시 실행할 수 있는가”입니다. 따라서 강사는 기능 시연을 끝낸 뒤 반드시 재현 질문을 던져야 합니다. “내일 혼자 한다면 첫 줄에 무엇을 쓰겠습니까?”, “결과가 이상하면 무엇을 먼저 고치겠습니까?”, “사람이 확인할 기준은 어디에 적어두겠습니까?” 같은 질문이 좋습니다.

운영 장면강사가 해야 할 일놓치면 생기는 문제보정 멘트
도입 3분오늘 만들 산출물을 먼저 보여준다.수강생이 강의를 기능 소개로 받아들인다.“오늘은 많이 보는 수업이 아니라 하나를 남기는 수업입니다.”
첫 데모일부러 부족한 요청을 먼저 보여준다.수강생이 좋은 결과만 보고 자기 실패를 부끄러워한다.“첫 답이 부족해야 고칠 지점이 보입니다.”
개인 실습 시작예제를 그대로 베끼지 말고 단어 3개라도 자기 업무로 바꾸게 한다.교재 속 예시는 되지만 실제 업무 적용이 안 된다.“부서명, 대상, 결과물 이름만 바꿔도 자기 실습이 됩니다.”
중간 점검완성도를 보지 말고 빠진 조건 하나를 찾게 한다.빠른 수강생은 지루해지고 느린 수강생은 포기한다.“지금은 완성이 아니라 다음 질문 하나를 찾는 시간입니다.”
공유잘한 사례와 수정 사례를 하나씩 고른다.잘한 사람 발표만 남아 초보자가 위축된다.“수정 사례가 더 좋은 학습 자료가 됩니다.”
마무리내일 다시 쓸 첫 행동을 적게 한다.수업 만족은 높지만 실제 전이가 약하다.“내일 복사해 넣을 첫 문장을 남기고 끝내겠습니다.”

수준별 분기 운영

수강생 유형보이는 반응강사 개입추가 과제
완전 초보입력창 앞에서 멈추고 무엇을 써야 할지 묻는다.전체 문장을 요구하지 말고 목적, 대상, 형식 세 칸만 채우게 한다.교재 예제에서 자기 업무 단어 3개만 바꾼다.
업무 숙련자자기 방식이 분명해서 Copilot 답을 잘 믿지 않는다.검토 기준을 만들게 하여 전문성을 Copilot 사용법 안으로 끌어들인다.Copilot 답변의 위험 표현 3개를 찾아 고친다.
AI 경험자빠르게 결과를 만들지만 검증을 건너뛴다.추가 질문, 근거 확인, 출력 형식 개선을 요구한다.같은 요청을 3가지 형식으로 바꿔 결과 차이를 비교한다.
관리자개인 실습보다 조직 적용과 통제 기준을 묻는다.정책 토론으로 길게 빠지지 말고 사용 가능 업무와 금지 업무를 나누게 한다.팀 공통 프롬프트 규칙 5개를 만든다.

강사의 관찰 포인트

  • 손이 멈춘 사람: 이해가 부족한 것이 아니라 첫 업무 선택이 너무 클 수 있습니다. “보고서 작성”을 “보고서 목차 만들기”처럼 줄여줍니다.
  • 결과를 바로 복사하는 사람: 속도가 빠른 것이 아니라 검토 습관이 빠졌을 수 있습니다. 숫자, 날짜, 사람 이름, 책임 표현을 형광펜으로 표시하게 합니다.
  • 질문이 많은 사람: 방해가 아니라 전이 가능성이 높은 신호입니다. 질문을 “지금 실습에서 확인할 것”과 “조직 정책으로 확인할 것”으로 분리합니다.
  • 화면이 다른 사람: 기능 차이를 길게 설명하지 말고 입력창 중심 대체 경로로 안내합니다. 오늘의 목표가 버튼 암기가 아니라 업무 지시와 검토라는 점을 반복합니다.

현장 리커버리 스크립트

상황 “제 답변은 선생님 화면이랑 너무 다른데요.”
대응 “좋습니다. Copilot 수업에서 같은 답이 나오는 것보다 중요한 것은 같은 기준으로 고치는 것입니다. 지금 답에서 목적, 대상, 형식, 확인 기준 중 무엇이 빠졌는지 보겠습니다.”
상황 “이건 그냥 제가 쓰는 게 더 빠른 것 같아요.”
대응 “맞는 업무도 있습니다. 오늘의 목표는 모든 일을 AI로 바꾸는 것이 아니라, 초안이 도움이 되는 업무와 사람이 바로 하는 편이 나은 업무를 구분하는 것입니다.”
상황 “회사 자료를 넣어도 되나요?”
대응 “수업에서는 민감자료를 넣지 않습니다. 실제 회사 적용은 조직 계정, 보안 정책, 자료 등급을 확인한 뒤 판단해야 합니다. 오늘은 비식별 예제로 같은 절차를 연습하겠습니다.”

강사 회고와 다음 수업 연결

수업이 끝난 뒤 강사는 결과물 개수만 세지 말고 어떤 조건에서 수강생이 멈췄는지 기록해야 합니다. 프롬프트 첫 문장을 못 쓰는지, 결과를 검토하지 못하는지, 화면 차이에 당황하는지, 보안 질문에서 흐름이 끊기는지에 따라 다음 수업의 도입 멘트가 달라집니다.

다음 장으로 넘어가기 전에는 이 장의 산출물을 버리지 말고 재사용한다고 예고합니다. Thinkervis판의 장점은 각 챕터가 따로 떨어진 지식이 아니라, 이전 산출물이 다음 실습의 재료가 되는 구조입니다. 수강생에게 “오늘 만든 것을 다음 장에서 다시 씁니다”라고 말하면 저장, 이름 붙이기, 정리 습관도 자연스럽게 생깁니다.

운영 실패 대비

상황 수강생의 Copilot 화면이 강사와 다르다.
대응 버튼 위치 설명을 멈추고 Copilot Chat 대체 흐름으로 전환한다. “원리는 같습니다. 본문을 붙여넣고 범위를 지정해봅시다”라고 안내한다.
상황 결과가 이상하게 나온다.
대응 실패로 처리하지 말고 “좋은 사례입니다. 어떤 정보가 빠졌는지 찾아봅시다”라고 말하며 프롬프트 개선 활동으로 바꾼다.
상황 일부 수강생이 너무 빨리 끝내고, 일부는 시작도 못 한다.
대응 빠른 수강생에게는 같은 프롬프트를 더 짧게, 더 공손하게, 더 표 형식으로 바꾸는 확장 과제를 줍니다. 늦은 수강생에게는 업무 범위를 줄여 예제 문장을 자기 단어 3개만 바꾸게 합니다.
상황 보안이나 저작권 질문이 나와 흐름이 끊긴다.
대응 즉석 토론으로 길게 가지 말고 “민감자료는 넣지 않는다, 원문 출처를 확인한다, 최종 책임은 사람이 진다”는 세 문장으로 잠시 정리한 뒤 실습으로 돌아옵니다.

AX_standard 보강: HTML 배포와 발표자료까지 연결하기

AX_standard에는 HTML MVP를 만드는 것에서 끝나지 않고, 다른 사람이 볼 수 있게 공유하거나 발표자료처럼 활용하는 단계까지 포함됩니다. 강사는 여기서 “배포는 멋있어 보이기 위한 작업”이 아니라 “내 PC에서만 보이던 시안을 회의 참석자가 함께 보는 상태로 바꾸는 작업”이라고 설명합니다.

공유 방식언제 쓰나주의할 점
HTML 파일 직접 전송가장 빠르게 내부 동료에게 시안을 보여줄 때받는 사람이 파일을 내려받아 열어야 합니다.
GitHub Pages더미 데이터 기반 공개 시안, 모바일 확인, 링크 공유가 필요할 때사내 데이터, 고객 정보, 미공개 실적은 절대 올리지 않습니다.
로컬 서버회의실 같은 같은 네트워크 환경에서 잠시 시연할 때PC 전원이 꺼지면 접근이 중단됩니다.
