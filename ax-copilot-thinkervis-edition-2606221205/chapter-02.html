챕터 2. 프롬프트 병아리반 — Copilot에게 심부름 잘 시키기

프롬프트는 어려운 주문서가 아닙니다. 쉽게 말해 ‘업무 심부름 메모’입니다. 사람에게 심부름을 시킬 때도 무엇을, 왜, 누구에게, 어떤 모양으로 해달라는지 알려줘야 하듯 Copilot에게도 같은 정보가 필요합니다.

나쁜 부탁과 좋은 부탁

“보고서 써줘”라는 말은 너무 넓습니다. Copilot은 보고서를 만들 수는 있지만, 누구에게 보고하는지, 왜 보고하는지, 얼마나 길어야 하는지, 어떤 내용을 빼야 하는지 모릅니다. 그래서 결과가 일반적이고 길어지거나, 반대로 중요한 내용이 빠질 수 있습니다.

좋은 프롬프트에는 보통 여섯 가지가 들어갑니다. 목적, 배경, 대상 독자, 결과물 형식, 제약조건, 검토 기준입니다. 처음부터 여섯 가지를 모두 넣으려고 하면 부담스럽습니다. 병아리반에서는 먼저 목적·대상·형식 세 가지만 넣어도 충분히 좋아집니다.

나쁜 예:
이 자료 보고서로 만들어줘.

좋은 예:
아래 자료를 팀장에게 공유할 1페이지 보고서 초안으로 정리해줘.
목적은 의사결정 준비이고, 형식은 핵심 요약 / 근거 / 리스크 / 확인 질문으로 나눠줘.
원문에 없는 내용은 추정이라고 표시해줘.
[병아리] 프롬프트를 길게 쓰면 시간이 더 걸리지 않나요?
처음에는 그렇습니다. 하지만 자주 하는 업무는 템플릿으로 저장해두면 됩니다. 매번 새로 긴 문장을 쓰자는 뜻이 아니라, 반복 업무에 쓸 좋은 심부름 메모를 만들어두자는 뜻입니다.

따라하기: 내 업무 프롬프트 만들기

수강생에게 종이나 메모장을 열게 한 뒤, 자기 업무 하나를 적게 합니다. 예를 들어 “주간보고 작성”, “고객 문의 답변”, “회의록 정리”, “교육 안내문 작성”처럼 매주 반복되는 일이 좋습니다.

  1. 먼저 업무 이름을 씁니다.
  2. 이 결과물을 볼 사람을 씁니다.
  3. 원하는 결과물 모양을 씁니다.
  4. 절대 하면 안 되는 일을 씁니다.
  5. 확인해야 할 질문을 요청합니다.
강사 “여러분의 업무는 특수해서 AI가 못 할 수도 있습니다. 괜찮습니다. 업무 전체를 맡기지 말고, 읽기·정리·초안 만들기 중 한 단계만 맡겨봅시다.”
수강생 예상 “제 업무는 너무 특수해서 예제가 안 맞는 것 같아요.”
강사 답변 “그럴수록 업무 전체가 아니라 작은 단계를 골라야 합니다. 예를 들어 판단은 사람이 하고, 자료 정리나 초안 작성만 Copilot에게 맡길 수 있습니다.”

프롬프트 고치기 연습

강사는 일부러 부족한 프롬프트를 보여주고 수강생에게 고치게 해야 합니다. 이것이 가장 효과적인 훈련입니다.

부족한 프롬프트무엇이 빠졌나요?고친 예
메일 답장 써줘상대, 톤, 약속 가능 범위협력사 담당자에게 정중히 답장하되, 일정 단축을 확정 약속하지 말고 내부 검토 후 회신하겠다고 써줘.
데이터 분석해줘분석 질문, 기준, 검증문의 유형 TOP 5, 평균 처리시간이 긴 제품군, 재문의율이 높은 항목을 계산 기준과 함께 정리해줘.
요약해줘목적, 형식, 후속 질문회의 준비용으로 핵심 주장, 근거, 리스크, 회의 질문 3개로 나눠 정리해줘.

Thinkervis판 수업 설계 노트

이 장의 운영 목표는 수강생이 챕터 2. 프롬프트 병아리반 — Copilot에게 심부름 잘 시키기를 “이해했다”고 말하는 데서 끝나지 않고, 자기 업무에 적용 가능한 산출물 하나를 남기는 것입니다. 따라서 강사는 설명, 데모, 개인 실습, 짝 토론, 공유 순서를 명확히 운영해야 합니다. 기능 설명이 10분을 넘으면 수강생의 손이 멈춥니다. 설명은 짧게, 실습은 작게, 피드백은 자주 주는 구조가 좋습니다.

구간권장 시간강사 행동완료 기준
개념 열기5분업무 상황 하나로 문제를 제시한다.수강생이 왜 필요한지 말할 수 있다.
데모7분부족한 프롬프트와 개선 프롬프트를 비교한다.결과 품질 차이를 눈으로 확인한다.
개인 실습12분자기 업무로 바꾸게 한다.개인 프롬프트 초안 1개가 나온다.
공유6분2명 정도의 사례를 공개 피드백한다.다른 업무에도 적용 가능하다고 느낀다.

평가 루브릭

  • 1점: Copilot에게 단순히 “해줘”라고만 입력한다.
  • 2점: 업무 목적이나 대상 중 하나를 포함한다.
  • 3점: 목적, 대상, 결과물 형식을 포함한다.
  • 4점: 검토 기준과 주의사항까지 포함한다.
  • 5점: Copilot 결과를 검토하고 추가 질문으로 개선한다.

운영 실패 대비

상황 수강생의 Copilot 화면이 강사와 다르다.
대응 버튼 위치 설명을 멈추고 Copilot Chat 대체 흐름으로 전환한다. “원리는 같습니다. 본문을 붙여넣고 범위를 지정해봅시다”라고 안내한다.
상황 결과가 이상하게 나온다.
대응 실패로 처리하지 말고 “좋은 사례입니다. 어떤 정보가 빠졌는지 찾아봅시다”라고 말하며 프롬프트 개선 활동으로 바꾼다.
