챕터 7. 강사 지도서 — 수업 전날부터 마무리까지

좋은 자료가 있어도 강사가 준비하지 않으면 수업은 흔들립니다. 이 장은 강사가 수업 전날, 수업 시작, 실습 중, 마무리에서 무엇을 말하고 확인해야 하는지 정리한 지도서입니다.

수업 전날 체크리스트

  • 강의 계정으로 Copilot 접속 확인
  • 파일 첨부 가능 여부 확인
  • Outlook, Excel, Word 안 Copilot 버튼 표시 여부 확인
  • 첨부가 안 될 때 사용할 붙여넣기용 샘플 준비
  • 개인정보가 없는 샘플 데이터 준비
  • 화면 공유 시 글자 크기 확인
  • 인터넷이 느릴 때 설명으로 대체할 예비 흐름 준비
잠깐만요 │ 강사는 실패 장면도 준비해야 합니다

Copilot이 예상과 다른 답을 할 때가 있습니다. 이때 당황해서 새로고침만 반복하면 수강생도 불안해집니다. 오히려 ‘좋습니다. 이런 답이 나올 때 어떻게 고치는지 보겠습니다’라고 말하면 훌륭한 학습 장면이 됩니다.

마무리 과제

수업 끝에는 “오늘 배운 기능”을 묻지 말고 “내일부터 바꿔볼 업무 하나”를 쓰게 해야 합니다. 예를 들어 주간보고 초안, 회의록 정리, 고객 문의 분류, 교육 안내문 작성처럼 작고 반복되는 일이 좋습니다.

내일부터 Copilot으로 바꿔볼 업무 하나를 정리해줘.
1. 업무 이름
2. 지금 걸리는 시간
3. Copilot에게 맡길 단계
4. 사람이 반드시 확인할 단계
5. 첫 프롬프트 초안
형식은 표로 작성해줘.
강사 “오늘의 목표는 Copilot 기능을 많이 아는 것이 아니라, 내 업무 하나를 Copilot과 함께 시작할 수 있게 되는 것입니다.”
수강생 예상 “오늘 배운 걸 다 기억 못 할 것 같아요.”
강사 답변 “괜찮습니다. 전부 기억할 필요 없습니다. 내 업무 하나, 프롬프트 템플릿 하나만 가져가도 성공입니다.”

Thinkervis판 수업 설계 노트

이 장의 운영 목표는 수강생이 챕터 7. 강사 지도서 — 수업 전날부터 마무리까지를 “이해했다”고 말하는 데서 끝나지 않고, 자기 업무에 적용 가능한 산출물 하나를 남기는 것입니다. 따라서 강사는 설명, 데모, 개인 실습, 짝 토론, 공유 순서를 명확히 운영해야 합니다. 기능 설명이 10분을 넘으면 수강생의 손이 멈춥니다. 설명은 짧게, 실습은 작게, 피드백은 자주 주는 구조가 좋습니다.

구간권장 시간강사 행동완료 기준
개념 열기5분업무 상황 하나로 문제를 제시한다.수강생이 왜 필요한지 말할 수 있다.
데모7분부족한 프롬프트와 개선 프롬프트를 비교한다.결과 품질 차이를 눈으로 확인한다.
개인 실습12분자기 업무로 바꾸게 한다.개인 프롬프트 초안 1개가 나온다.
공유6분2명 정도의 사례를 공개 피드백한다.다른 업무에도 적용 가능하다고 느낀다.

평가 루브릭

  • 1점: Copilot에게 단순히 “해줘”라고만 입력한다.
  • 2점: 업무 목적이나 대상 중 하나를 포함한다.
  • 3점: 목적, 대상, 결과물 형식을 포함한다.
  • 4점: 검토 기준과 주의사항까지 포함한다.
  • 5점: Copilot 결과를 검토하고 추가 질문으로 개선한다.

운영 실패 대비

상황 수강생의 Copilot 화면이 강사와 다르다.
대응 버튼 위치 설명을 멈추고 Copilot Chat 대체 흐름으로 전환한다. “원리는 같습니다. 본문을 붙여넣고 범위를 지정해봅시다”라고 안내한다.
상황 결과가 이상하게 나온다.
대응 실패로 처리하지 말고 “좋은 사례입니다. 어떤 정보가 빠졌는지 찾아봅시다”라고 말하며 프롬프트 개선 활동으로 바꾼다.
수업 설계 점검표:
1. 이 장에서 수강생이 남길 산출물은 무엇인가?
2. 강사 데모는 7분 안에 끝나는가?
3. 화면이 다를 때 대체 경로가 있는가?
4. 수강생의 결과물을 평가할 기준이 있는가?
5. 다음 장과 연결되는 한 문장이 준비되어 있는가?
