챕터 6. HTML MVP 병아리반 — 아이디어를 화면으로 보여주기

HTML MVP 실습은 코딩 수업이 아닙니다. 말로만 설명하던 아이디어를 한 화면으로 보여주고, 사람들이 같은 그림을 보며 이야기하게 만드는 수업입니다.

이 장을 시작하기 전에 보는 수업 지도

오늘 남길 것업무 아이디어를 입력, 처리, 출력 화면으로 나눈 HTML MVP 설계안
사람이 확인할 것예쁜 화면보다 사용자가 실제로 무엇을 입력하고 얻는지 확인한다.
강사 운영 포인트설명 20%, 시연 30%, 수강생 자기 업무 변환 50%로 운영합니다.

왜 HTML을 만들까

회의에서 “고객 이슈 대시보드가 필요합니다”라고 말하면 사람마다 다른 그림을 떠올립니다. 어떤 사람은 표를, 어떤 사람은 그래프를, 어떤 사람은 보고서를 생각합니다. Copilot에게 단일 HTML 초안을 만들게 하면 모두가 같은 화면을 보며 이야기할 수 있습니다.

단일 HTML 파일로 회의용 대시보드 초안을 만들어줘.
주제는 고객 VOC 우선순위 정리입니다.
포함할 것: 상단 요약 카드 3개, 문의 유형 TOP 5 표, 우선 조치가 필요한 항목, 다음 액션 목록.
디자인은 깔끔하고 업무용으로, 한 파일 안에 CSS까지 포함해줘.
실제 개인정보는 넣지 말고 샘플 데이터로 작성해줘.
[병아리] HTML을 모르는 사람도 할 수 있나요?
네. 이 실습의 목표는 HTML 문법을 외우는 것이 아닙니다. Copilot에게 원하는 화면을 설명하고, 나온 결과를 열어보며 업무에 맞게 수정 요청하는 것입니다.

수업 중 자주 생기는 문제

첫째, 코드가 너무 길어져서 수강생이 겁을 먹을 수 있습니다. 이때 강사는 “코드를 읽는 시간이 아니라 결과 화면을 보고 요구사항을 고치는 시간”이라고 말해야 합니다. 둘째, 브라우저에서 파일을 여는 방법을 모를 수 있습니다. 파일을 .html로 저장하고 더블클릭하면 된다고 천천히 안내하세요. 셋째, 디자인이 마음에 들지 않을 수 있습니다. 이때는 색상, 여백, 카드 수, 표 컬럼처럼 구체적으로 수정 요청하게 합니다.

tip

HTML MVP는 완성품이 아니라 대화용 초안입니다. 강사는 ‘이걸 배포하자’보다 ‘이 화면을 보고 어떤 의사결정을 할 수 있나요?’라고 물어야 합니다.

Hermes판 더 친절한 강의 노트

이 장을 가르칠 때 강사는 먼저 수강생의 마음을 낮춰야 합니다. Copilot을 처음 배우는 사람은 “이걸 모르면 뒤처지는 것 아닐까”라는 불안과 “그래도 내 업무는 내가 더 잘 알지”라는 방어를 동시에 가집니다. 그래서 설명은 기능 이름보다 익숙한 업무 장면에서 시작해야 합니다. 챕터 6. HTML MVP 병아리반 — 아이디어를 화면으로 보여주기라는 주제를 말하기 전에, 수강생이 어제 실제로 했을 법한 일 하나를 떠올리게 하세요.

강사는 빠르게 정답을 보여주기보다, 일부러 작은 시행착오를 남겨두는 편이 좋습니다. 첫 프롬프트는 조금 부족하게 넣고, 결과를 함께 보며 “무엇을 더 알려주면 좋아질까요?”라고 묻습니다. 이 순간 수강생은 Copilot을 마법 상자가 아니라 대화하며 다듬는 업무 파트너로 이해하기 시작합니다.

강사 “이번 장은 외우는 시간이 아닙니다. 여러분이 이미 하고 있는 업무를 Copilot에게 어떻게 설명하면 좋을지 같이 연습하는 시간입니다.”
수강생 예상 “제가 정확히 뭘 입력해야 할지 모르겠어요.”
강사 답변 “괜찮습니다. 처음부터 좋은 프롬프트를 쓰는 사람은 거의 없습니다. 먼저 거칠게 부탁하고, 나온 답을 보며 목적·대상·형식·주의사항을 하나씩 보태면 됩니다.”

수업 중 바로 쓸 확장 스크립트

“지금 Copilot이 준 답을 정답처럼 보지 말고, 신입 동료가 가져온 초안처럼 봅시다. 잘한 부분은 살리고, 빠진 배경은 추가하고, 위험한 표현은 고치면 됩니다. 오늘 우리가 배우는 것은 AI에게 일을 던지는 법이 아니라, AI와 함께 업무 초안을 다듬는 법입니다.”

Hermes 추가 보강: 초보 강사가 그대로 읽는 친절한 진행안

이 장은 설명을 많이 하는 것보다 수강생이 “아, 내 업무도 이렇게 작게 시작하면 되겠구나”라고 느끼게 만드는 것이 중요합니다. 강사는 어려운 용어를 먼저 꺼내지 말고, 수강생이 매일 보는 문서, 메일, 표, 회의 장면에서 출발합니다. 그런 다음 Copilot에게 맡길 수 있는 작은 작업과 사람이 확인할 기준을 나눠 보여줍니다.

“오늘 이 장에서 우리가 남길 것은 지식이 아니라 업무 아이디어를 입력, 처리, 출력 화면으로 나눈 HTML MVP 설계안입니다. 화면을 똑같이 따라 하는 것보다, 내 업무를 작게 설명하고 나온 결과를 의심하며 고치는 연습이 더 중요합니다.”

“답이 너무 평범해요.”“그럼 Copilot에게 더 알아야 할 조건을 알려줘야 합니다.”대상 독자, 분량, 말투, 금지사항을 추가한다.
“제 화면은 달라요.”“괜찮습니다. 오늘은 버튼 위치보다 요청 문장의 구조를 배웁니다.”Copilot Chat 또는 메모장 붙여넣기 흐름으로 대체한다.
“그냥 제가 하는 게 빠른데요.”“맞습니다. 모든 일을 맡기는 수업이 아닙니다.”초안, 요약, 질문 생성처럼 시작 시간이 줄어드는 업무만 고른다.

이 장의 7분 미니 실습

  1. 1분: 오늘 다룰 업무 상황을 하나 고릅니다.
  2. 2분: Copilot에게 처음 부탁할 문장을 아주 거칠게 씁니다.
  3. 2분: 결과에서 빠진 조건을 하나 찾습니다.
  4. 1분: 조건을 추가해 다시 요청합니다.
  5. 1분: 사람이 확인할 항목을 표시합니다.

AX_standard 보강: HTML 배포와 발표자료까지 연결하기

AX_standard에는 HTML MVP를 만드는 것에서 끝나지 않고, 다른 사람이 볼 수 있게 공유하거나 발표자료처럼 활용하는 단계까지 포함됩니다. 강사는 여기서 “배포는 멋있어 보이기 위한 작업”이 아니라 “내 PC에서만 보이던 시안을 회의 참석자가 함께 보는 상태로 바꾸는 작업”이라고 설명합니다.

공유 방식언제 쓰나주의할 점
HTML 파일 직접 전송가장 빠르게 내부 동료에게 시안을 보여줄 때받는 사람이 파일을 내려받아 열어야 합니다.
GitHub Pages더미 데이터 기반 공개 시안, 모바일 확인, 링크 공유가 필요할 때사내 데이터, 고객 정보, 미공개 실적은 절대 올리지 않습니다.
로컬 서버회의실 같은 같은 네트워크 환경에서 잠시 시연할 때PC 전원이 꺼지면 접근이 중단됩니다.

조건:
1. 한 화면에 한 메시지만 보이게 섹션을 나눠줘.
2. 현상 / 원인 / 대응 / 기대효과 구조를 유지해줘.
3. 표가 필요한 부분은 HTML table로 정리해줘.
4. CSS와 JS는 모두 단일 HTML 안에 포함해줘.
5. 실제 사내 데이터가 아니라 더미 데이터임을 화면 하단에 표시해줘.
