챕터 6. HTML MVP 병아리반 — 아이디어를 화면으로 보여주기

HTML MVP 실습은 코딩 수업이 아닙니다. 말로만 설명하던 아이디어를 한 화면으로 보여주고, 사람들이 같은 그림을 보며 이야기하게 만드는 수업입니다.

왜 HTML을 만들까

회의에서 “고객 이슈 대시보드가 필요합니다”라고 말하면 사람마다 다른 그림을 떠올립니다. 어떤 사람은 표를, 어떤 사람은 그래프를, 어떤 사람은 보고서를 생각합니다. Copilot에게 단일 HTML 초안을 만들게 하면 모두가 같은 화면을 보며 이야기할 수 있습니다.

단일 HTML 파일로 회의용 대시보드 초안을 만들어줘.
주제는 고객 VOC 우선순위 정리입니다.
포함할 것: 상단 요약 카드 3개, 문의 유형 TOP 5 표, 우선 조치가 필요한 항목, 다음 액션 목록.
디자인은 깔끔하고 업무용으로, 한 파일 안에 CSS까지 포함해줘.
실제 개인정보는 넣지 말고 샘플 데이터로 작성해줘.
[병아리] HTML을 모르는 사람도 할 수 있나요?
네. 이 실습의 목표는 HTML 문법을 외우는 것이 아닙니다. Copilot에게 원하는 화면을 설명하고, 나온 결과를 열어보며 업무에 맞게 수정 요청하는 것입니다.

수업 중 자주 생기는 문제

첫째, 코드가 너무 길어져서 수강생이 겁을 먹을 수 있습니다. 이때 강사는 “코드를 읽는 시간이 아니라 결과 화면을 보고 요구사항을 고치는 시간”이라고 말해야 합니다. 둘째, 브라우저에서 파일을 여는 방법을 모를 수 있습니다. 파일을 .html로 저장하고 더블클릭하면 된다고 천천히 안내하세요. 셋째, 디자인이 마음에 들지 않을 수 있습니다. 이때는 색상, 여백, 카드 수, 표 컬럼처럼 구체적으로 수정 요청하게 합니다.

tip

HTML MVP는 완성품이 아니라 대화용 초안입니다. 강사는 ‘이걸 배포하자’보다 ‘이 화면을 보고 어떤 의사결정을 할 수 있나요?’라고 물어야 합니다.

Thinkervis판 수업 설계 노트

이 장의 운영 목표는 수강생이 챕터 6. HTML MVP 병아리반 — 아이디어를 화면으로 보여주기를 “이해했다”고 말하는 데서 끝나지 않고, 자기 업무에 적용 가능한 산출물 하나를 남기는 것입니다. 따라서 강사는 설명, 데모, 개인 실습, 짝 토론, 공유 순서를 명확히 운영해야 합니다. 기능 설명이 10분을 넘으면 수강생의 손이 멈춥니다. 설명은 짧게, 실습은 작게, 피드백은 자주 주는 구조가 좋습니다.

구간권장 시간강사 행동완료 기준
개념 열기5분업무 상황 하나로 문제를 제시한다.수강생이 왜 필요한지 말할 수 있다.
데모7분부족한 프롬프트와 개선 프롬프트를 비교한다.결과 품질 차이를 눈으로 확인한다.
개인 실습12분자기 업무로 바꾸게 한다.개인 프롬프트 초안 1개가 나온다.
공유6분2명 정도의 사례를 공개 피드백한다.다른 업무에도 적용 가능하다고 느낀다.

평가 루브릭

  • 1점: Copilot에게 단순히 “해줘”라고만 입력한다.
  • 2점: 업무 목적이나 대상 중 하나를 포함한다.
  • 3점: 목적, 대상, 결과물 형식을 포함한다.
  • 4점: 검토 기준과 주의사항까지 포함한다.
  • 5점: Copilot 결과를 검토하고 추가 질문으로 개선한다.

운영 실패 대비

상황 수강생의 Copilot 화면이 강사와 다르다.
대응 버튼 위치 설명을 멈추고 Copilot Chat 대체 흐름으로 전환한다. “원리는 같습니다. 본문을 붙여넣고 범위를 지정해봅시다”라고 안내한다.
상황 결과가 이상하게 나온다.
대응 실패로 처리하지 말고 “좋은 사례입니다. 어떤 정보가 빠졌는지 찾아봅시다”라고 말하며 프롬프트 개선 활동으로 바꾼다.
